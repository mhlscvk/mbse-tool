# 2. Canonical Artifact Hierarchy

To prevent confusion between source files, parsed models, validation results, and rendered diagrams, the system defines the following artifact hierarchy.

## 2.1 Source Revision (Authoritative Artifact)

The **authoritative artifact** of the system is always the `.sysml` source file.

Properties:

- Stored as revisioned text
- Immutable after commit
- Associated with user identity
- Associated with timestamp
- Associated with project and file

Every change to a model creates a **new source revision**.

---

## 2.2 Parsed Model Snapshot

A **Parsed Model Snapshot** is a structured representation derived from a specific source revision.

Properties:

- Generated by the parser
- Stored per revision
- Versioned with parser version
- Used by validation and rendering

This snapshot enables:

- Deterministic validation
- Faster rendering
- Reproducible diagnostics

---

## 2.3 Validation Result Set

Validation produces a **diagnostic result set**.

Properties:

- Tied to a specific parsed model snapshot
- Tied to a specific validator version
- Immutable after generation
- Used by UI and AI tooling

Diagnostics include:

- Issue ID
- Severity
- Message
- Location
- Category
- Related model element

---

## 2.4 Render View Model

The **Render View Model** is generated from a parsed model snapshot.

Properties:

- Not authoritative
- Cached for performance
- Regenerated when necessary

The graphical viewer must **never depend directly on source text**.

---

## 2.5 AI Patch Proposal

AI systems must never modify models directly.

Instead they produce a **Patch Proposal**.

Properties:

- Contains proposed changes
- Linked to a specific source revision
- Validated before commit
- Requires user approval

If approved, a **new source revision** is created.

# 3. Parser Execution Model

Parsing behavior must be deterministic and predictable.

---

## 3.1 Parse On Save

Parsing occurs automatically when:

- A file is uploaded
- A file is saved
- An AI patch is accepted

---

## 3.2 Parse Job Lifecycle

Each parse request creates a **Parse Job**.

Lifecycle states:

- PENDING
- RUNNING
- COMPLETED
- FAILED
- SUPERSEDED

A job is marked **SUPERSEDED** when a newer revision replaces it.

---

## 3.3 Synchronous vs Asynchronous Parsing

Phase 1 parsing strategy:

### Small Models

Parsed synchronously.

Expected parse time:

Less than 1 second.

### Large Models

Parsed asynchronously using a background worker.

The UI must indicate when parsing is in progress.

---

## 3.4 Parse Supersession

If a new revision appears before parsing finishes:

The previous parse job becomes **SUPERSEDED**.

Only the **latest successful parse result** should be used by validation and rendering.

---

# 4. Phase 1 Graphical Viewer Scope

The graphical viewer in Phase 1 must remain intentionally limited.

---

## 4.1 Viewer Purpose

Provide a **read-only structural visualization** of supported SysML constructs.

---

## 4.2 Supported Features

Phase 1 viewer supports:

- Block definitions
- Basic relationships
- Simple hierarchical structures

---

## 4.3 Unsupported Features

Phase 1 viewer will NOT support:

- Diagram editing
- Graphical round-trip editing
- Custom layouts
- Advanced diagram types
- Full SysML coverage

Unsupported elements must be clearly indicated in the UI.

---

## 4.4 Layout Strategy

Layout uses a **basic automatic layout engine**.

Manual diagram layout is deferred to later phases.

---

# 6. Permission Matrix

Phase 1 defines three roles.

| Capability | Admin | Standard User | Viewer |
|------------|------|---------------|--------|
| Create project | Yes | Yes | No |
| Edit model | Yes | Yes | No |
| View model | Yes | Yes | Yes |
| Delete project | Yes | No | No |
| Manage billing | Yes | Yes | No |
| Create MCP tokens | Yes | Yes | No |
| Approve AI patch | Yes | Yes | No |

Authorization must be enforced by a **central authorization module**.

---

# 7. Normalized Model Persistence Strategy

Artifacts are stored as follows.

| Artifact | Persisted | Notes |
|--------|-----------|------|
| Source revision | Yes | Authoritative |
| Parsed model snapshot | Yes | Per revision |
| Validation results | Yes | Stored with metadata |
| Render view model | No | Cached only |
| AI patch proposal | Yes | Requires approval |

---

# 8. Concurrency Model

Multiple saves may occur during editing.

The system must handle this safely.

---

## 8.1 Revision Creation

Each save generates a **new source revision**.

---

## 8.2 Parse Queue Behavior

Workflow:

Save → Revision Created → Parse Job Created

If a newer revision appears:

Old parse job becomes **SUPERSEDED**.

---

## 8.3 Viewer Behavior

While parsing:

The viewer displays the **last valid rendered model**.

After parsing completes:

The viewer automatically refreshes.

---

# 9. Observability Strategy

Model-processing workflows must be observable.

The system must log key lifecycle events.

---

## 9.1 Model Lifecycle Events

- MODEL_UPLOAD_ACCEPTED
- REVISION_CREATED
- PARSE_STARTED
- PARSE_COMPLETED
- PARSE_FAILED
- VALIDATION_COMPLETED
- RENDER_GENERATED

---

## 9.2 AI Tool Events

- MCP_TOOL_INVOKED
- AI_PATCH_PROPOSED
- AI_PATCH_VALIDATED
- AI_PATCH_APPROVED
- AI_PATCH_REJECTED

---

## 9.3 System Events

- AUTH_LOGIN
- AUTH_FAILED
- BILLING_EVENT_RECEIVED
- SYSTEM_ERROR

All logs should use **structured logging**.

---

# 10. Testing Strategy

Testing must be organized by module.

---

## 10.1 Parser Tests

Parser testing uses **fixture-based tests**.

Each fixture contains:

- Input SysML
- Expected normalized model
- Expected diagnostics

---

## 10.2 Validation Tests

Each validation rule must include:

- Valid examples
- Failing examples
- Edge cases

---

## 10.3 Render Tests

Viewer outputs should use **snapshot tests**.

Input: normalized model  
Output: render view model

---

## 10.4 API Tests

API testing must include:

- Contract tests
- Authentication tests
- Permission tests

---

## 10.5 MCP Tool Tests

AI tools must be tested for:

- Authorization
- Audit logging
- Patch validation
